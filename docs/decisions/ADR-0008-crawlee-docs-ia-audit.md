# ADR-0008: Crawlee documentation IA audit and Millipede site contract

## 1. Status and context

Status: Accepted for Phase 9.

Phase 9, specified by `docs/ROADMAP.md` lines 348–381, builds Millipede's documentation site with Next.js, `fumadocs-ui`, and `fumadocs-mdx`. Crawlee's JavaScript and Python Docusaurus sites are reference corpora for structure, page inventory, and the pedagogical sequence from quick start through a first crawler, concepts, guides, and examples. Their prose is not a source: all site wording must be original to Millipede and grounded in Millipede's code, crate READMEs, and `docs/guide/`.

The Python documentation was written second and corrects several information-architecture problems in the JavaScript documentation. Where the two structures disagree, the Python structure wins unless this ADR records a Millipede-specific reason to do otherwise.

The existing `docs/guide/*.md` files remain untouched during this phase. Replacing them with pointers is a later lead decision after Phase 8 closes; the site ports their material without changing the originals.

## 6. Snippet-integrity decision

Every Rust snippet rendered on the site is generated from a source Cargo actually compiles:

1. Whole files under `millipede/examples/*.rs`, compiled by `cargo build --examples`.
2. The single `rust` fence in `millipede/README.md` and in each `crates/*/README.md`. Each crate's `lib.rs` includes its README through `#![doc = include_str!(...)]`, so these fences are doc-tested. Extraction strips rustdoc hidden lines beginning with `# `.

Fences in `docs/guide/*.md` are not doc-tested and are therefore never snippet sources.

`scripts/sync-snippets.mjs` writes committed MDX partials beneath `website/snippets/examples/` and `website/snippets/readme/`. Content pages consume those partials with Fumadocs `<include>`. In `--check` mode the script regenerates the complete expected set in memory and byte-compares it with disk. A single comparison catches drift in either direction and fails for stale, missing, or orphan partials. It also rejects every hand-written `rust` or `rs` fence under `website/content/docs/**`.

CI runs `--check` only. Write mode never runs in CI, so regeneration cannot conceal stale committed output.

Raw inclusion of `.rs` files at render time is rejected because it provides no explicit CI integrity gate, has fragile path semantics, and does not reliably participate in development watch mode. If Fumadocs `<include>` regresses, the recorded fallback is JSX-comment marker injection into the consuming pages while retaining the same generated partials and byte-comparison gate.

Guides whose teaching material has no compiled source, including sitemap crawling, configuration, and architecture, are deliberately fence-free and link to docs.rs instead. Fully mechanizing new teaching snippets is deferred until new doc-tested sources may be added after Phase 8.

## 7. Deployment decision

Use GitHub Pages when deployment is enabled. The site is fully static through Next.js `output: 'export'` and a static search index, so GitHub Pages, Vercel, and Cloudflare Pages are equivalent for required capabilities. GitHub Pages requires no new vendor account, token, or DNS setup; deployment is a single workflow job the lead can enable; and hosting costs nothing at any plausible project traffic level.

Resolve static-host constraints in the application from the start:

- Generate a static search index.
- Read `basePath` and `assetPrefix` from `NEXT_PUBLIC_BASE_PATH`: empty for local development and CI, `/millipede` on GitHub Pages.
- Set `trailingSlash: true`.
- Set `images.unoptimized: true`.

The accepted trade-off is the absence of per-pull-request preview deployments. Vercel is the runner-up because it provides preview deployments with nearly zero configuration; revisit it if Millipede gains a custom documentation domain. Cloudflare Pages adds no capability the static site currently needs.

The decision is enforced in code, not merely recorded: `.github/workflows/website.yml` contains a `deploy-pages` job that implements it. The job runs only on push to `main`, requires the `build` job (snippet-staleness, build, and internal-link gates) to pass first, rebuilds the site with `NEXT_PUBLIC_BASE_PATH=/millipede`, and publishes `website/out` through `actions/upload-pages-artifact` and `actions/deploy-pages` under the `github-pages` environment with job-scoped `pages: write` / `id-token: write` permissions. `actions/configure-pages` with `enablement: true` creates the Pages site on the first run, so no manual repository-settings step precedes the first deployment. The job uses its own `cancel-in-progress: false` concurrency group so an in-flight deployment is never cancelled by a newer push.

No deployment occurs within Phase 9 itself: the job triggers exclusively on push to `main`, and landing this workflow on `main` is the lead's commit. Merging Phase 9 therefore executes the deployment decision automatically; the first push to `main` after merge publishes the site.

## 8. Link policy

Internal links are CI-blocking. Linkinator scans the exported `out/` tree with recursion confined to the local site root; it must never recursively crawl third-party hosts.

External URLs receive single, bounded status probes in a separate `continue-on-error` step. This includes docs.rs because crate pages may return 404 until version 0.1.0 is live on crates.io. After publication the lead may make the external step blocking. Until then, prose links target docs.rs crate roots only, never deep item paths whose generated anchors or module layout could change at publish time.
